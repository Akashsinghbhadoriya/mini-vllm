# vLLM — Core Concepts

## Why vLLM?

Naive HuggingFace inference generates tokens one-by-one without any memory management or batching strategy. vLLM solves this and achieves up to **24x better throughput** by carefully managing GPU memory across multiple concurrent requests.

## Autoregressive Generation

LLMs generate text one token at a time. To produce token `t`, the model needs to attend over **all previous tokens** `0..t-1`. Recomputing those attention scores from scratch at every step is wasteful — so we cache them.

## KV Cache

During the attention computation, each token produces a **Key** and **Value** vector. These are stored in a cache (`past_key_values` in HuggingFace) and reused for all future decode steps.

Without KV cache: O(n²) compute per token (full recomputation).  
With KV cache: O(n) compute per token (only the new token is processed).

The catch: when serving many users simultaneously, KV caches for all active sequences must fit in GPU memory simultaneously. This blows up fast.

## Prefill vs Decode

Every request goes through two distinct phases:

| Property        | Prefill              | Decode               |
| --------------- | -------------------- | -------------------- |
| Input           | Full prompt (large)  | Last token only (1)  |
| Parallelism     | High (all tokens at once) | Low (one at a time) |
| GPU utilization | High (compute-bound) | Lower (memory-bound) |
| KV cache        | Created              | Reused               |
| Bottleneck      | Compute              | Memory bandwidth     |

**Prefill phase**: The user's prompt is tokenized and fed through the model in one forward pass. This creates the initial KV cache and produces the first output token.

**Decode phase**: Each subsequent token is generated by feeding only the last token to the model, along with the cached KV values. This continues until an EOS token or `max_new_tokens` is hit.

Almost all vLLM optimizations revolve around balancing and scheduling these two phases efficiently across many concurrent requests.

## Memory Problem

Each sequence's KV cache grows linearly with the number of generated tokens. With many concurrent users:
- KV caches compete for the same GPU memory
- Traditional approach allocates contiguous memory per sequence (wasteful)
- vLLM uses **PagedAttention** to solve this — see [`PagedAttention.md`](./PagedAttention.md)
